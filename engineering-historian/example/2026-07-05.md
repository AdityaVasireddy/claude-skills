# notes-api — 2026-07-05

## What I worked on
Goal for the session: get notes-api off my laptop and onto the cloud VM with a repeatable deploy, then wire GitHub Actions so a push to `main` redeploys automatically. Got both done, but the middle four hours were eaten by two problems that were entirely self-inflicted (see Mistakes). Ended the session with a green pipeline and a verified public endpoint.

## Decisions made

**1. Docker Compose on a single VM, not Kubernetes or a PaaS.**
Rejected Kubernetes: one service, one developer — k3s alone would cost more maintenance than the app. Rejected Railway/Render (PaaS): tried Railway for two days last month; cold starts on the free tier made the health endpoint flap, and debugging inside their build system was slower than owning the box. Rejected bare systemd services: workable, but Compose gives me the same `docker compose up -d` flow locally and on the VM, so dev/prod stay identical. Revisit trigger: if this ever needs a second node or zero-downtime deploys, re-evaluate — not before.

**2. nginx on the host as reverse proxy, not Caddy, not nginx-in-a-container.**
Rejected Caddy: automatic TLS is attractive, but the VM already had nginx installed and I know its config language; switching costs a learning session for zero user-visible gain. Rejected containerized nginx: certbot renewal hooks and host firewall rules get more complicated when the proxy is inside Compose. Kept it boring: host nginx → proxy_pass to the app container on 127.0.0.1:8000.

**3. Postgres in the same Compose file, data on a named volume — not managed DB.**
Rejected managed Postgres (~$15/mo): this is a personal tool with recoverable data; a nightly `pg_dump` to object storage (TODO below) is enough durability for now. The named volume decision matters: first attempt used a bind mount and permissions fought me (see Problems solved). Revisit trigger: any data I'd be upset to lose for more than a day.

**4. Deploy = SSH + `git pull` + `docker compose up -d --build` from GitHub Actions, not a registry push.**
Rejected pushing images to GHCR and pulling on the VM: an extra moving part (registry auth on the VM) for a single-machine deploy. Building on the VM is slower (~90s) but has one less credential to leak and one less thing to debug. Revisit trigger: build times over ~5 min or a second deploy target.

## Commands that worked

Open the firewall for HTTP/HTTPS on the VM (Ubuntu, ufw) — the app was unreachable until BOTH the cloud provider's security-list rule AND the on-host firewall allowed traffic; doing only one silently fails:

```bash
# on the VM
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw status verbose        # verify: 80,443 ALLOW IN
```

TLS certificate with certbot's nginx plugin (after DNS A record was pointing at the VM):

```bash
# on the VM
sudo certbot --nginx -d notes.example.dev --redirect -m me@example.dev --agree-tos
sudo certbot renew --dry-run   # verify auto-renewal works BEFORE forgetting about it
```

Deploy step that CI runs over SSH (idempotent, safe to re-run):

```bash
cd /opt/notes-api && git pull --ff-only && docker compose up -d --build && docker compose ps
```

Generate the dedicated deploy key (private half goes in GitHub repo secrets as `VM_SSH_KEY`, public half appended to the VM's `~/.ssh/authorized_keys`):

```bash
# on my laptop
ssh-keygen -t ed25519 -f deploy_key -C "gha-deploy-notes-api" -N ""
```

Verify end-to-end from outside the VM (the only test that counts):

```bash
curl -fsS https://notes.example.dev/health   # expect {"status":"ok"}
```

## Problems solved

**App unreachable from the internet despite running fine on the VM.**
Symptom: `curl localhost:8000/health` worked on the VM; from my laptop, connection timed out. Root cause: two firewalls in series — I had added the cloud provider's ingress rule but forgot the VM's own ufw, which was denying 80/443. Fix: ufw allow commands above. Lesson generalizes: on any cloud VM assume *provider security list AND host firewall* both need the port; check both before debugging the app.

**Postgres container crash-looping after switching to a bind mount.**
Symptom: `docker compose logs db` showed `initdb: could not change permissions of directory`. Root cause: bind-mounted host directory owned by my user; the postgres image runs as uid 999 and can't chown it on this filesystem. Fix: switched to a named volume (`pgdata:`) and let Docker own the storage. Rule of thumb recorded: named volumes for databases, bind mounts only for config files.

**GitHub Actions SSH step failing with `Host key verification failed`.**
Symptom: pipeline red at the deploy step, ran fine from my laptop. Root cause: the runner has an empty `known_hosts`; ssh refuses unknown hosts non-interactively. Fix: added a step that runs `ssh-keyscan -H $VM_HOST >> ~/.ssh/known_hosts` before the deploy command. Considered `StrictHostKeyChecking=no` and rejected it — it silences MITM warnings permanently to save one line of setup.

## Mistakes

- Debugged the "unreachable" problem for ~45 minutes at the nginx layer (rewriting proxy config, checking logs) before checking firewalls. The failure was below the layer I was staring at. Correction adopted: when a service is unreachable, walk the stack bottom-up — DNS → provider firewall → host firewall → listener → proxy → app — instead of starting at the layer I most recently touched.
- Committed the first Compose file with the Postgres password inline. Caught it before pushing, moved it to `.env` (gitignored) — but the correct order was `.env` *first*. Correction: secrets file is now the first file created in any new project, before any service config exists.
- Ran the certbot command before the DNS A record had propagated; certbot's HTTP challenge failed and I briefly suspected nginx. Ten minutes lost to a problem that was just "wait." Correction: `dig +short notes.example.dev` must return the VM IP before running certbot, every time.

## Questions I still have

- Rate limiting: nginx `limit_req` vs doing it in FastAPI middleware — where does it belong when there's exactly one proxy and one app?
- Is `--ff-only` on the deploy pull too strict? A force-push to main would brick deploys until someone SSHes in. Acceptable failure mode or footgun?
- Backup strategy: is a nightly `pg_dump` cron on the VM enough, or does the dump need to leave the machine to count as a backup at all? (It does. The real question is the cheapest way to ship it off-box.)
- Should CI run the test suite as a gate before the deploy step, or is deploy-on-green-main discipline enough for a solo project?
